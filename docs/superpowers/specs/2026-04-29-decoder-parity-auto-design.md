# 协议分析器自动修复设计

**目标**

在不暂停向用户确认的前提下，围绕当前仓库已有的 7 个协议分析器，按固定顺序做可验证的 parity 修复，并把执行边界、验证方式和完成标准固化到文档与测试中。

**范围**

- 公共底座：
  - 解码器注册真值来源
  - 协议输出模型的最小补强
  - golden / 单测断言强化
- 协议顺序：
  - `I2C`
  - `SPI`
  - `UART`
  - `CAN`
  - `LIN`
  - `I2S`
  - `Streaming I2C`

**设计决策**

1. 使用项目根目录 `.worktree/decoder-parity-auto-20260429/` 作为唯一实现工作区。
2. 每个协议只交给一个独立 subagent 实现，协议之间串行，不并发修改。
3. 每个协议都遵循同一循环：
   - 先补一个失败测试
   - 验证失败
   - 做最小修复
   - 跑协议定向测试
   - 主线程做规范复核和代码复核
4. 公共底座修改只允许服务于当前协议所需的最小能力，避免一次性重构整个解码器框架。
5. 完成标准按 P0 优先：只要当前协议的 P0 缺口被测试证明关闭，就允许进入下一个协议；P1/P2 仅在不阻塞整体串行推进时顺带完成。

**实现顺序**

1. 文档与计划落盘
2. `I2C`
3. `SPI`
4. `UART`
5. `CAN`
6. `LIN`
7. `I2S`
8. `Streaming I2C`
9. 统一回归验证与剩余项收口

**验证策略**

- 单协议阶段：
  - 协议专属单测
  - `DecoderParityCore.test.ts`
  - `DecoderSigrokGolden.test.ts`
- 公共变更阶段：
  - 影响范围内的解码器测试集合
  - 需要时补 `webview/main.test.ts` 或 host 相关定向测试
- 收尾阶段：
  - 至少跑一遍所有 decoder 相关测试集合

**风险控制**

- 如果框架级改动会同时影响多个协议，优先把最小必要接口抽出来，并在当前协议里先落测试。
- 如果某个协议的 P0 明显超出当前轮可控范围，则保留剩余项到主文档，不阻塞后续协议启动。
- 如果公共底座修改导致其它协议回归失败，优先修复回归，再继续串行推进。
