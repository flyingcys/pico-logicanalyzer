# 协议分析插件闭环与手工测试准入评审

分析日期：2026-04-29  
仓库：`/home/share/samba/vscode-extension/pico-logicanalyzer`  
评审目标：判断当前已开发协议分析插件是否可以进入手工测试阶段，并列出进入手工测试前必须闭环的工作。  
约束口径：不新增协议分析插件；当前已经存在的旧协议能力必须闭环，不能继续堆新协议数量掩盖旧协议缺口。

## 1. 最终结论

当前项目不能进入正式手工测试阶段。

可以进行的只有开发者探索性自测，例如打开 `.lac`、观察波形、用 CH0/CH1 固定映射跑 I2C happy path。该类探索不能写成“手工测试通过”，也不能写入真实硬件认证矩阵作为通过证据。

阻断原因不是单点问题，而是闭环链路同时存在 5 个 P0 缺口：

| 阻断项 | 结论 | 证据 |
| --- | --- | --- |
| 协议单测门禁 | 失败 | `npm run test:decoders -- --runInBand`：5 个套件中 3 个失败，146 个测试中 4 个失败 |
| Quick CI 门禁 | 失败 | `npm run test:ci:quick -- --skip-install`：14 个核心测试文件中 1 个失败，失败文件为 `I2CDecoder.test.ts` |
| I2C 主链路 | 未闭环 | 逆序命名通道只输出 Start，不输出地址、ACK、Stop；声明的 bit 注释不产出 |
| 注册口径 | 未闭环 | `DecoderRegistry.registerAllDecoders()` 不注册 `i2c/spi/uart`，但 `getDecoderRegistryInfo()` 又声明它们存在 |
| VSIX 交付包 | 未闭环 | `package:dry` 返回 0，但 `vsce ls` 把 `.worktree/decoder-parity-auto-20260429` 及其中大量文件列入包清单 |

因此，当前最合理的推进策略是：冻结协议数量，先修旧协议闭环；在 P0 全部通过前，不开发 USB、JTAG/SWD 或其它新协议分析插件。

## 3. 手工测试准入标准

本项目的“进入手工测试阶段”至少应满足以下条件。没有同时满足时，只能算开发自测或调试验证。

| 维度 | 最低准入标准 | 当前状态 |
| --- | --- | --- |
| 自动化门禁 | `typecheck`、`typecheck:strict`、`lint`、`test:decoders`、`test:ci:quick` 全部通过 | 未通过 |
| VSIX 交付 | `package:dry` 产物清单干净，不包含 `.worktree`、源码、测试、node_modules 副本 | 未通过 |
| 协议入口 | 被声明支持的旧协议有清晰入口，或明确标为不参与本轮手工测试 | 未通过 |
| I2C 主链路 | UI、VSCode host、DecoderManager、I2CDecoder 行为一致 | 未通过 |
| 测试模板 | 手工测试记录模板覆盖操作步骤、预期结果、失败记录、附件 hash | 部分具备，仅 I2C 模板存在 |
| 认证口径 | fixture、mock、HTML 模拟、真实硬件结果分开记录 | 部分具备 |

当前可以保留 I2C UI 的探索性检查，但正式手工测试准入应至少等待 P0 清单闭环。

## 4.1 7 个可独立运行任务拆分

统一约束：

- worktree 均创建在项目根目录 `.worktree` 下，目录名固定为 `01` 到 `07`。
- 每个任务先补或修正能证明缺口的自动化测试，再修改实现。
- 不新增第 8 个协议，不把上层协议或设备级协议纳入本轮。
- 手工测试候选不等于真实硬件认证通过；真实硬件结果必须另写测试记录、日志、截图和 `.lac` hash。

### 4.1.1 01 I2C 任务说明

范围：

- 只修改 I2C、注册入口和必要的 I2C UI/host 适配。
- 修复 `[SDA, SCL]` 或命名逆序输入时仍硬编码 `captureIndex: 0/1` 的问题。
- 对完整地址字节和数据字节产出 bit annotation，保持 annotation row 声明和实际输出一致。
- 让 `DecoderRegistry.registerAllDecoders()`、`getDecoderRegistryInfo()` 和 Manager 真实注册结果一致。

完成定义：

- I2C 专项测试通过。
- `test:decoders` 中当前 I2C 失败清零。
- 第一轮手工测试范围可明确写为 I2C UI 主链路候选。

### 4.1.2 02 Streaming I2C 任务说明

范围：

- 不单独扩大协议数量；只处理 `StreamingI2CDecoder` 与常规 I2C 的闭环关系。
- 若作为透明执行模式，`executeDecoder('i2c')` 在大样本下切换到流式实现，但输出契约必须与常规 I2C 一致。
- 若保留为独立 `streaming_i2c`，文档和 UI 必须明确它不代表 I2C 主链路已经大样本闭环。
- 串行化 I2C 分块处理，避免跨块全局状态在并发下乱序。

完成定义：

- 跨块地址、数据、ACK/NACK、Stop 结果顺序稳定。
- annotationType、地址格式、rawData 与常规 I2C 对齐或有明确差异声明。
- 有停止请求和大样本边界测试。

### 4.1.3 03 SPI 任务说明

范围：

- 优先闭环协议实现、Manager 路径和选项复用问题。
- 修复 decoder 实例缓存导致的 CPOL/CPHA、bit order、word size、CS 极性选项泄漏。
- 明确 `putCSChange()` 的短期处理：实现可测输出，或从可测矩阵中降级删除。
- 若接入 UI，提供协议选择、CLK/MISO/MOSI/CS 映射和核心选项表单。

完成定义：

- 直接实例化、Manager 执行和 golden 三条路径结果一致。
- 连续两次不同 SPI 配置运行不互相污染。
- 手工测试模板能覆盖至少一个 MOSI/MISO 传输、CS 极性和 CPHA/CPOL 组合。

### 4.1.4 04 UART 任务说明

范围：

- 优先闭环协议实现、Manager 路径和选项复用问题。
- 明确逐通道 `decodeChannelSimple()` 与 wait/edge/idle 主路径的关系。
- 将 break、idle、frame warning 纳入主路径，或从能力矩阵中明确降级。
- 若接入 UI，提供 RX/TX 映射、baudrate、data bits、parity、stop bits、invert、sample point 选项。

完成定义：

- RX/TX 双向事件顺序可测试。
- 连续两次不同 UART 配置运行不互相污染。
- packet delimiter/packet length 若不实现，必须在文档中标为不支持。

### 4.1.5 05 CAN 任务说明

范围：

- 目标限定为 classic CAN，不做 CAN-FD。
- 将单帧理想位流推进为持续扫描多帧。
- 实现 bit stuffing 去填充和基础采样重同步。
- CRC 可先保持区段读取，但必须在能力矩阵中标明“未真实校验”，除非本任务同步补齐 CRC 校验。

完成定义：

- 多帧、stuff bit、错误边界均有专项测试。
- 真实波形级 golden 至少覆盖一个连续帧样例。
- 未实现 CAN-FD、真实 CRC 校验等能力不得写入手工测试通过项。

### 4.1.6 06 LIN 任务说明

范围：

- 先确定 LIN 主路径是 raw logic 解码还是 UART 级联；实现必须与文档一致。
- 补多帧生命周期：break、sync、PID、data、checksum、无响应。
- 修正 checksum 选项与 LIN version 语义。
- 增加 LIN 专项单测，不再只依赖汇总 golden。

完成定义：

- 多帧、主机头、从机响应、无响应、PID parity 错误、checksum 错误均可自动化验证。
- 如未实现 UART 级联，必须在功能矩阵中明确 LIN 仍为 raw logic 路径。
- 手工测试模板能区分正常响应、无响应和校验错误。

### 4.1.7 07 I2S 任务说明

范围：

- 按 WS 翻转闭合左右声道字边界，不再只按固定 `word_length` 截断。
- 修复 32 位样本使用 JS 位运算导致的符号位风险。
- 补首字同步、多余 bit warning、变字长边界和实例复用测试。
- binary/WAV 输出如不实现，必须在矩阵中标注为不支持。

完成定义：

- 16/24/32 位样本均有专项测试。
- 32 位最高位为 1 时数值语义正确。
- 左右声道顺序、WS 边界和多余 bit warning 可自动化验证。

## 5. 项目级闭环问题

### 5.1 注册入口分裂

当前存在三套口径：

| 入口 | 真实行为 |
| --- | --- |
| `DecoderManager.registerBuiltinDecoders()` | 构造时注册 `i2c/spi/uart/can/lin/i2s` 和 `streaming_i2c` |
| `src/decoders/index.ts initializeDecoders()` | 再次注册 `i2c/spi/uart/can/lin/i2s` |
| `DecoderRegistry.registerAllDecoders()` | 只注册 `can/lin/i2s/streaming_i2c` |
| `DecoderRegistry.getDecoderRegistryInfo()` | 硬编码返回 `i2c/spi/uart/can/lin/i2s` |

直接影响：

- 测试已暴露失败：`registerAllDecoders 会真实注册常规 i2c` 不通过。
- 文档、注册信息和运行时行为不是同一个真值来源。
- 后续如果 UI 根据 registry 构建协议列表，会出现“列表显示支持，实际入口没注册”或“运行时支持，列表不可信”的问题。

闭环要求：

- 只保留一个权威注册来源。
- `registerAllDecoders()`、`getAvailableDecoders()`、`getDecoderRegistryInfo()` 必须来自同一份数据。
- 注册测试不能只断言硬编码数组，必须验证 manager 真实状态。

### 5.2 输出模型没有闭环

`DecoderOutputType` 已声明 `ANNOTATION/PYTHON/BINARY/LOGIC/META`，但 `DecoderBase.put()` 当前把所有输出折叠为 `DecoderResult`，只保留 `annotationType/values/rawData/shape`。

影响：

- I2C、SPI、UART 与 sigrok/logicanalyzer 的 Python packet、binary、metadata 输出无法真正对齐。
- LIN 级联 UART、上层协议消费下层输出的能力只停留在 `addOutput/getInput` 的局部机制，没有形成 UI 可见、测试可验的输出契约。
- 即使某个协议 annotation 可见，也不能声明完整 parity。

闭环要求：

- 明确短期手工测试只验 annotation，还是必须验 Python/Binary/Meta。
- 如果只验 annotation，功能矩阵必须降级表述，不能写成完整协议 parity。
- 若目标是原版闭环，必须扩展结果模型和 UI 展示/导出路径。

### 5.3 前端主路径只暴露 I2C

现行 Vue3 主界面右侧面板只支持 I2C：

- `decoderStore.activeDecoderConfigs` 固定为 `[{ decoderId: 'i2c', label: 'I2C' }]`。
- `runI2CDecoder()` 固定发送 `decoderId: 'i2c'`。
- `AppSidebarRight.vue` 标题、映射控件、按钮文案全部固定为 I2C。
- HTML/browser host 中 `runBrowserDecoder()` 对非 `i2c` 直接返回 `Decoder not found`。

VSCode host 侧 `LACEditorProvider.runDecoderForDocument()` 可以按任意 `decoderId` 调 `DecoderManager.executeDecoder()`，但 UI 没有选择入口，也没有为 SPI/UART/CAN/LIN/I2S 做通道映射和选项配置。

闭环要求：

- 如果本轮只手工测试 I2C，必须明确其它旧协议“不进入手工测试”，而不是写成用户可测。
- 如果要求旧协议全部手工闭环，则需要协议选择、动态通道映射、动态选项、结果行标签、错误提示、测试模板全部覆盖。

### 5.4 VSIX 清单污染

`.vscodeignore` 没有排除 `.worktree/`。由于项目约束要求 worktree 均放在项目根目录 `.worktree` 下，这个目录会长期存在。

本次 `npm run package:dry` 的 `vsce ls` 输出包含：

- `.worktree/decoder-parity-auto-20260429/...`
- `.worktree/decoder-parity-auto-20260429/node_modules/...`
- `.worktree/decoder-parity-auto-20260429/src/...`
- `.worktree/decoder-parity-auto-20260429/tests/...`
- `.worktree/decoder-parity-auto-20260429/docs/...`

直接影响：

- 手工测试安装包体积、内容、加载时间都不可控。
- 可能把历史 worktree 的旧源码、旧文档、旧测试一起交付。
- 任何 VSIX smoke 结果都不可信，因为包不是干净候选。

闭环要求：

- `.vscodeignore` 必须显式排除 `.worktree/`。
- 重新运行 `package:dry` 并人工抽查清单。
- 包清单只能包含运行所需 `out/`、`package.json`、`README.md`、`CHANGELOG.md` 等必要文件。

### 5.5 测试策略仍偏宽松

当前 `sigrok-golden.json` 每个协议只有 normal/error/boundary 各 1 个，共 18 个用例。大量断言使用 `arrayContaining/objectContaining`，只证明“包含某些片段”，不证明：

- 结果顺序正确。
- 样本区间正确。
- 没有多余结果。
- 错误边界正确。
- 连续多帧正确。
- 实例复用后选项不泄漏。
- UI host 与直接实例化结果一致。

闭环要求：

- P0 先让当前失败用例通过。
- P1 将 golden 从子集断言升级为关键事务的完整序列断言。
- 对每个旧协议增加至少一个通过 `DecoderManager.executeDecoder()` 的测试，覆盖实例缓存和通道映射。

### 6.1 I2C

判定：不准入正式手工测试。

已具备能力：

- 有现行 UI 入口。
- VSCode host 可通过 `runDecoder` 调用真实 `I2CDecoder`。
- HTML host 有模拟结果，可用于浏览器 smoke。
- 常规写、NACK、repeated start 等 golden 用例基础可跑。

P0 阻断：

- 逆序命名通道失败。`I2CDecoder.decode()` 先查找 `sclChannel/sdaChannel`，但后续 `prepareChannelData()` 仍硬编码 `captureIndex: 0/1`，导致 `[SDA, SCL]` 输入只输出 Start。
- 声明了 `bit` 注释和 `bits` annotation row，但 `handleAddressOrData()` 只缓存 bit，完整字节时直接输出地址或数据，没有发出 annotationType 5。
- `registerAllDecoders()` 未注册常规 `i2c`。
- 当前失败测试已经覆盖以上缺口，不能忽略。

P1 缺口：

- Python/Binary/Meta 输出缺失。
- 10 位地址路径未被足够测试。
- `warning` 注释声明但没有真实输出路径。
- `Read/Write` 位没有单独注释。
- HTML host 是合成结果，不是浏览器端真实解码。

进入手工测试前必须完成：

1. 修复通道映射，使用实际找到的 SCL/SDA 或 manager 映射后的通道。
2. 发出完整字节的 bit 注释，或删除声明并同步测试和文档。考虑原版对齐，应优先补发 bit 注释。
3. 修复 `registerAllDecoders()` 的真实注册。
4. `npm run test:decoders -- --runInBand` 通过。
5. `npm run test:ci:quick -- --skip-install` 通过。

### 6.2 Streaming I2C

判定：不准入手工测试，也不应作为 I2C 大样本能力对外声明。

已具备能力：

- 有独立 `StreamingI2CDecoder` 类。
- `DecoderManager.getRecommendedExecutionMode('i2c', > 1_000_000)` 会推荐 `streaming`。
- 流式基类支持分块、进度、部分结果和停止。

P0 阻断：

- 推荐 `streaming` 不等于实际执行 `streaming_i2c`。主 UI 仍调用 `runI2CDecoder()`，VSCode host 仍走 `executeDecoder('i2c')`。
- `StreamingI2CDecoder` 的 annotationType 与常规 I2C 不一致。常规 I2C 的 `address-read/address-write` 是 6/7，流式 I2C 使用 5/6。
- 地址格式逻辑与常规 I2C 相反。常规 I2C `shifted` 显示 7 位地址，流式 I2C `shifted` 显示 `currentByte`。
- 流式基类默认并发处理分块，`StreamingI2CDecoder` 又依赖 `globalState` 跨块状态。I2C 是强顺序协议，并发分块存在边界重复、乱序和状态污染风险。
- 没有直接协议行为测试、跨块边界测试、停止测试和 UI 接入测试。

闭环要求：

- 决定它是 `i2c` 的透明执行模式，还是单独协议 `streaming_i2c`。
- 若作为透明模式，`executeDecoder('i2c')` 需要按策略切换并保持输出契约一致。
- 禁止并发分块或重做有序状态传递。
- 与常规 I2C 统一 annotationType、address format、结果文案和 rawData。

### 6.3 SPI

判定：算法级可继续自动化补测，不准入用户手工测试。

已具备能力：

- `SPIDecoder.test.ts` 通过。
- `DecoderSigrokGolden.test.ts` 中 SPI normal/error/boundary 用例通过。
- 支持 CLK、MISO、MOSI、CS，支持 CPOL/CPHA、bit order、word size、CS 极性。
- 可输出 bit、data word 和 transfer annotation。

P0 阻断：

- 现行 UI 没有 SPI 入口。
- HTML host 明确拒绝非 I2C。
- VSCode host 虽然能按 `decoderId` 调用，但用户没有通道映射和选项配置入口。
- `putCSChange()` 是空实现，CS-CHANGE Python 输出没有闭环。

P1 缺口：

- 选项实例字段只在传入时覆盖，`DecoderManager` 又缓存 decoder 实例。连续两次运行不同配置时，未传入的选项可能沿用上次值。
- 非 annotation 输出缺失。
- 缺少经过 `DecoderManager.executeDecoder()` 的实例复用测试。

进入手工测试前必须完成：

1. 明确 SPI 是否进入本轮手工测试范围。
2. 若进入，新增 UI 协议选择、CLK/MISO/MOSI/CS 映射、选项配置。
3. 修复选项默认值重置和 CS change 输出契约。
4. 增加 Manager 路径测试和手工测试模板。

### 6.4 UART

判定：算法级可继续自动化补测，不准入用户手工测试。

已具备能力：

- `UARTDecoder.test.ts` 通过。
- `DecoderSigrokGolden.test.ts` 中 UART normal/error/boundary 用例通过。
- 支持 RX/TX、baudrate、data bits、parity、stop bits、bit order、format、invert、sample point。
- 可输出起始位、数据位、数据值、校验、停止位、frame warning。

P0 阻断：

- 现行 UI 没有 UART 入口。
- 主解码循环实际是逐通道 `decodeChannelSimple()`，不是完整的 wait/edge/idle 主路径。
- break/idle 相关函数存在，但没有真正接入主路径。

P1 缺口：

- 原版 packet delimiter/packet length 配置缺失。
- 当前 packet 行为硬编码 16 字节输出。
- RX/TX 双向事件顺序没有严格验证。
- Python/Binary 输出缺失。
- 与 LIN 的 UART 级联输出契约没有闭环。

进入手工测试前必须完成：

1. 明确是否作为本轮手工测试协议。
2. 若进入，新增 UI 映射和选项面板。
3. 补 packet 配置语义，或在功能矩阵中明确不支持 packet。
4. 将 break/idle/frame 行为纳入主路径或删除误导性声明。

### 6.5 CAN

判定：不准入手工测试。

已具备能力：

- 有 `CANDecoder` 类。
- golden 中 normal/error/boundary 用例通过。
- 可解析理想位流中的 SOF、标准/扩展 ID、DLC、数据、CRC 区段、ACK、EOF。

P0 阻断：

- 当前只从第一个 SOF 开始采样并解析一帧，没有持续扫描多帧。
- 没有 bit stuffing 去填充。
- 没有 dominant edge 重同步。
- 不支持真实 CAN 波形中的时钟漂移、采样点校准和连续帧。
- 无 UI 入口。

P1 缺口：

- 不支持 CAN-FD。
- CRC 仅作为区段读取，没有真实校验。
- 注释粒度和 Python 输出不对齐。

进入手工测试前必须完成：

1. 先完成 classic CAN 真实位流状态机。
2. 补 bit stuffing、重同步、连续多帧。
3. 增加真实波形级 golden，再考虑 UI 手工测试。

### 6.6 LIN

判定：不准入手工测试。

已具备能力：

- 有 `LINDecoder` 类。
- golden 中 normal/error/boundary 用例通过。
- 可从单线 raw logic 中解析 break、sync、PID、data、checksum。
- 能做 PID parity 和 checksum 检查。

P0 阻断：

- 输入模型与原版不一致。当前声明 `inputs = ['logic']`，原版 LIN 通常叠在 UART 事件流之上。
- 当前只解析一个帧，不能持续处理多帧。
- 强依赖 `data_length`，缺少主机头、从机响应、无响应等生命周期。
- `findBreak()` 返回第一个低电平段，真实波形中容易误判。
- 无 UI 入口。

P1 缺口：

- checksum 选项与原版 version 语义不一致。
- 没有 UART 到 LIN 的级联测试。
- 没有独立 LIN 单测文件，主要靠汇总 golden。

进入手工测试前必须完成：

1. 先决定 LIN 是否必须改成 UART 级联模型。
2. 补多帧生命周期和主机头/响应/无响应分支。
3. 增加 LIN 专项单测和 UI 手工测试模板。

### 6.7 I2S

判定：不准入手工测试。

已具备能力：

- 有 `I2SDecoder` 类。
- golden 中 normal/error/boundary 用例通过。
- 支持 SCK/WS/SD，支持 `word_length` 和 `justification`。
- 可输出左右声道样本 annotation。

P0 阻断：

- 当前按固定 `word_length` 截断字，原版更接近按 WS 翻转界定字边界。
- 首字同步策略不完整。
- 32 位样本使用 JS 位运算累积，存在符号位风险。
- 多余 bit 被静默丢弃。
- 无 UI 入口。

P1 缺口：

- Python 输出、binary/WAV 输出缺失。
- 没有独立 I2S 单测文件。
- 没有实例复用和 32 位高位为 1 的测试。

进入手工测试前必须完成：

1. 改为按 WS 翻转闭合字边界。
2. 修复 32 位数值语义。
3. 补首字同步、变字长 warning、32 位边界、实例复用测试。
4. 若进入 UI 手工测试，新增 SCK/WS/SD 映射和音频样本结果展示。

## 9. 不应继续开发的新协议

在 P0 和 P1 未完成前，不建议开发以下新协议：

- USB low/full speed。
- JTAG/SWD。
- I2C 派生过滤器或设备级协议。
- SPI Flash、SD card、EEPROM 等上层协议。
- 任何新的“第一版 TypeScript 状态机”协议。

原因：当前旧协议还没有形成从注册、执行、UI、测试、打包、手工记录到认证矩阵的闭环。继续新增协议只会扩大未闭环面积。

## 10. 推荐推进顺序

建议按以下顺序推进，不要并行修改太多协议实现：

1. 修 I2C 和注册 P0，让当前失败测试全部通过。
2. 修 `.vscodeignore`，确保 `package:dry` 清单干净。
3. 重新跑 `typecheck`、`typecheck:strict`、`lint`、`test:decoders`、`test:ci:quick`、`test:webview:unit`、`test:webview:smoke`、`package:dry`。
4. 只把 I2C 标为“可进入手工测试候选”，其它协议标为“算法存在，UI 未准入”。
5. 完成 I2C 手工测试模板中的真实 `.lac`、截图、日志、hash 记录。
6. 再决定是否扩展 UI 到 SPI/UART。
7. CAN/LIN/I2S 先补协议语义和测试，不要先接 UI。
