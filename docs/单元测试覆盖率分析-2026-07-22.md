## 二、本次清理：移除 webview 死代码

### 背景
项目存在两套前端代码：`src/webview/`（旧）与 `src/frontend/`（新）。经核查确认插件实际加载的是 `src/frontend`：

```
webpack.config.js (webviewConfig)
  entry: src/frontend/app/main-vscode.ts  ← 真实入口
       + src/frontend/app/main-html.ts
  → 打包到 out/webview/main-vscode.js
LACEditorProvider.getHtmlForWebview → 加载 out/webview/*
```

`out/webview` 只是输出目录名，内容全是 `src/frontend` 打的包。

### 死代码判定证据
1. **全仓零引用**：`grep -rn "src/webview"` 在 `src/`、`tests/`、`webpack.config.js` 中除自身外无任何源码 import。
2. **GitNexus impact**：`webview/utils/LayoutManager` 上游调用 = **0，0 进程，LOW 风险（exact）**。
3. **孤儿闭环**：`webview/engines/` 9 个渲染引擎（~5000 行）互相引用成岛，外部零入口，不进 webpack bundle，插件不加载。
4. **测试测错对象**：`tests/unit/webview/` 下多数测试实际 import 的是 `src/frontend/`，仅 `LayoutManager.resource.test.ts` 真测 webview 死代码。

### 清理操作
| 操作 | 对象 |
|------|------|
| 删除目录 | `src/webview/`（27 文件：App.vue + 8 组件 + 10 engines + 测试 + i18n + main + shims + utils） |
| 删除测试 | `tests/unit/webview/LayoutManager.resource.test.ts` |
| 改配置 | `tsconfig.json` 移除 `"src/webview/shims-vue.d.ts"` |
| 保留 | webpack `webviewConfig`（entry=frontend，是 frontend 构建链）、`package.json` 构建脚本、`jest.webview.config.js` 及 `tests/unit/webview/` 其余 4 个测试（实测 frontend，目录名为历史遗留） |

### 验证
- `npm run typecheck`：**exit 0** ✓
- `jest --config jest.webview.config.js`：**PASS 100 / FAIL 0** ✓
- 覆盖率分母从 20118 行降至 18051 行（-2067 行可执行代码）。

## 八、结论：是否需要提升覆盖

**总体判断：需要，但应精准投入，而非追求整体百分比。**

1. **不需要补的部分**
   - `src/webview/` 死代码已删——这本身就是覆盖率分析的最大收益（消除误导）。
   - 协议解码器主体（I2C/SPI/CAN/UART/LIN/I2S）已达 87%–99%，是项目质量基线，维持即可。
   - `driver-sdk/templates`、`examples`、`testing`：模板/示例/工具代码，单测价值低，可用快照或集成方式验证。

2. **应当补的部分（按优先级）**
   - **P0 核心路径必须补**：`WaveformRenderer`、`LACEditorProvider`、`extension.ts`、`SigrokAdapter`、`HardwareCompatibilityDatabase`。这些是用户可感知的运行时行为，bug 直接表现为功能失效，ROI 最高。
   - **P1 按需补**：解码周边（ChannelMapping/PerformanceOptimizer）、数据压缩、硬件驱动适配器。

3. **建议设置覆盖率门禁**
   - 当前无 `coverageThreshold`，建议至少对 `src/models`、`src/services`、`src/decoders/protocols` 设 80% 下限，防止核心层退化。
   - 全局不必强设高门禁（会逼迫给 SDK 模板刷无意义测试）。

4. **driver-sdk 需产品决策**
   - 若 driver-sdk 是面向第三方驱动开发者的战略功能 → 补 tools（ProtocolHelper/HardwareCapabilityBuilder/DriverValidator）单测，至少 60%。
   - 若仅为内部脚手架 → 维持现状，补少量集成测试即可。
