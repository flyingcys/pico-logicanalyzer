## 待验证记录入口

真实设备记录统一追加到 `docs/hardware-validation/records/`。当前目录只提供 pending 模板，不包含任何真实设备通过结论。只有记录中同时包含具体设备、固件版本、commit、采集配置、结果文件 `sha256`、截图或日志路径，并且结论为 `pass` 时，才允许回填本矩阵并提升对应设备状态。

## 验证记录模板

每次真实设备验证必须追加以下信息：

| 字段 | 内容 |
| --- | --- |
| 日期 | YYYY-MM-DD |
| 操作系统 | Windows / macOS / Linux 版本 |
| 设备型号和序列号 | 真实型号、固件版本、序列号可脱敏 |
| 扩展版本和 commit | VSIX 版本、Git commit |
| 命令或操作路径 | VSCode 命令、CLI 命令或测试脚本 |
| 采集配置 | 频率、pre/post、通道、触发 |
| 结果文件 | `.lac`、CSV、日志、截图路径 |
| 结论 | pass / fail / blocked / pending，并链接问题 |

## I2C UI 手工验证入口

真实 Pico + I2C 目标设备的 UI 闭环记录使用 `docs/hardware-validation/i2c-pico-ui-manual-test.md`。在该模板完成前，I2C UI 只能标注为 fixture 或 experimental。

当前 I2C UI 证据只覆盖软件路径：fixture `.lac` 可打开、波形可显示、右侧 I2C 解码 UI 可通过 `runDecoder` 路径输出 START/ACK/STOP。真实设备缺口包括 Pico/Pico W/Pico 2 的设备型号与固件记录、实际 I2C 总线采集、结果文件 hash、波形截图、解码面板截图和结论。

## I2S UI 手工验证入口

真实 Pico + I2S 信号源的 UI 闭环记录使用 `docs/hardware-validation/i2s-pico-ui-manual-test.md`。在该模板完成前，I2S UI 只能标注为 fixture 或软件手工测试候选。

当前 I2S UI 证据覆盖软件路径：右侧协议解码面板可选择 I2S、配置 SCK/WS/SD、设置 `word_length` 和 `justification`，并通过 VSCode host `runDecoder('i2s')` 进入真实 `DecoderManager.executeDecoder()`。HTML host 的 `I2S HTML 模拟` 仅用于浏览器 smoke，不作为真实解码证据。I2S 当前只声明 annotation 样本输出，binary 输出和 WAV 导出不支持。

## 验证记录 Schema

数据库字段位置：`DeviceCompatibilityEntry.testStatus.certificationEvidence`。本节描述内部字段映射到文档证据等级的口径，当前文档只使用 Beta、fixture、framework、experimental、pending。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `evidenceLevel` | string | 是 | 内部证据字段；对外文档映射为 Beta、fixture、framework、experimental 或 pending。 |
| `records[].evidenceId` | string | 是 | 稳定记录 ID。 |
| `records[].date` | `YYYY-MM-DD` | 是 | 执行验证或 fixture 生成日期。 |
| `records[].operatingSystem` | string | 是 | 真实系统版本；fixture 可写 `CI fixture`。 |
| `records[].deviceModel` | string | 是 | 设备型号。 |
| `records[].firmwareVersion` | string | 真实设备必填 | 固件版本或握手返回版本。 |
| `records[].serialNumber` | string | 真实设备建议 | 可脱敏。 |
| `records[].extensionVersion` | string | 建议 | VSIX 或 package 版本。 |
| `records[].commit` | string | 真实设备必填 | Git commit。 |
| `records[].commandOrPath` | string | 是 | 命令、VSCode 操作路径或测试脚本。 |
| `records[].captureConfig` | object | 采集证据必填 | `sampleRateHz`、`preTriggerSamples`、`postTriggerSamples`、`channels`、`trigger`、`mode`。 |
| `records[].resultFiles[]` | object[] | 是 | 路径、文件类型和 hash；真实设备通过证据必须至少有一个带 `sha256` 的结果文件。 |
| `records[].result` | `pass` / `fail` / `blocked` / `pending` | 是 | 验证结论。 |
| `downgradeReason` | string | 条件必填 | 状态被降级时记录原因。 |

## 证据等级与降级规则

| 证据等级 | 含义 | 允许的最高文档等级 |
| --- | --- | --- |
| `framework` | 仅代码路径、接口或静态声明；不证明设备可采集 | `experimental` |
| `fixture` | 有协议 fixture、golden 或离线回放；不含真实设备采集 | `fixture` |
| `Beta` | 真实设备运行并保存可追溯结果文件 hash | 按记录结果提升，当前无通过记录 |

规则：

- 任何真实设备通过声明必须至少有一条结果为 `pass` 的真实设备记录，并包含系统、设备型号、固件版本、commit、采集配置、结果文件 hash。
- 缺少真实设备通过证据时，数据库校验会降级；有 fixture 通过记录时降为 `fixture`，否则降为 `experimental`。
- 普通 Raspberry Pi Pico 这类只被 VID/PID 命中的设备必须保持 `pending`，直到版本握手证明运行逻辑分析器固件并补充真实设备记录。
- README、用户手册和发布说明只能引用本矩阵中已有结果为 `pass` 的真实设备记录。
