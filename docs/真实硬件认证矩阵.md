## 认证记录模板

每次真实硬件认证必须追加以下信息：

| 字段 | 内容 |
| --- | --- |
| 日期 | YYYY-MM-DD |
| 操作系统 | Windows / macOS / Linux 版本 |
| 设备型号和序列号 | 真实型号、固件版本、序列号可脱敏 |
| 扩展版本和 commit | VSIX 版本、Git commit |
| 命令或操作路径 | VSCode 命令、CLI 命令或测试脚本 |
| 采集配置 | 频率、pre/post、通道、触发 |
| 结果文件 | `.lac`、CSV、日志、截图路径 |
| 结论 | 通过 / 失败 / 阻塞，并链接问题 |

## I2C UI 手工验证入口

真实 Pico + I2C 目标设备的 UI 闭环记录使用 `docs/hardware-validation/i2c-pico-ui-manual-test.md`。在该模板完成前，I2C UI 只能标注为 fixture 或实验性验证，不得提升为 `hardware`、`verified` 或 `certified`。

当前 I2C UI 证据只覆盖软件路径：fixture `.lac` 可打开、波形可显示、右侧 I2C 解码 UI 可通过 `runDecoder` 路径输出 START/ACK/STOP。真实硬件缺口包括 Pico/Pico W/Pico 2 的设备型号与固件记录、实际 I2C 总线采集、结果文件 hash、波形截图、解码面板截图和认证结论。

## 认证记录 Schema

数据库字段位置：`DeviceCompatibilityEntry.testStatus.certificationEvidence`。

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `evidenceLevel` | `mock` / `fixture` / `hardware` | 是 | 当前条目最高证据等级。 |
| `records[].evidenceId` | string | 是 | 稳定记录 ID。 |
| `records[].date` | `YYYY-MM-DD` | 是 | 执行认证或 fixture 生成日期。 |
| `records[].operatingSystem` | string | 是 | 真实系统版本；fixture 可写 `CI fixture`。 |
| `records[].deviceModel` | string | 是 | 设备型号。 |
| `records[].firmwareVersion` | string | 真实硬件必填 | 固件版本或握手返回版本。 |
| `records[].serialNumber` | string | 真实硬件建议 | 可脱敏。 |
| `records[].extensionVersion` | string | 建议 | VSIX 或 package 版本。 |
| `records[].commit` | string | 真实硬件必填 | Git commit。 |
| `records[].commandOrPath` | string | 是 | 命令、VSCode 操作路径或测试脚本。 |
| `records[].captureConfig` | object | 采集证据必填 | `sampleRateHz`、`preTriggerSamples`、`postTriggerSamples`、`channels`、`trigger`、`mode`。 |
| `records[].resultFiles[]` | object[] | 是 | 路径、文件类型和 hash；真实硬件通过证据必须至少有一个带 `sha256` 的结果文件。 |
| `records[].result` | `pass` / `fail` / `blocked` / `pending` | 是 | 认证结论。 |
| `downgradeReason` | string | 条件必填 | 状态被降级时记录原因。 |

## 证据等级与降级规则

| 证据等级 | 含义 | 允许的最高认证等级 |
| --- | --- | --- |
| `mock` | 仅 mock、代码路径或静态声明；不证明设备可采集 | `experimental` 或 `community` |
| `fixture` | 有协议 fixture、golden 或离线回放；不含真实设备采集 | `fixture` |
| `hardware` | 真实设备运行并保存可追溯结果文件 hash | `verified` 或 `certified` |

规则：

- `certified` 和 `verified` 必须至少有一条 `hardware + pass` 记录，并包含系统、设备型号、固件版本、commit、采集配置、结果文件 hash。
- 缺少真实硬件通过证据时，数据库校验会把 `certified/verified` 降级；有 fixture 通过记录时降为 `fixture`，否则降为 `experimental` 或 `community`。
- 普通 Raspberry Pi Pico 这类只被 VID/PID 命中的设备必须保持 `candidate`，直到版本握手证明运行逻辑分析器固件。
- README、用户手册和发布说明只能引用本矩阵中 `hardware` 证据已经通过的真实硬件能力。
