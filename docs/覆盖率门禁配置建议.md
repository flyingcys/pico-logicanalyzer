# 覆盖率门禁配置建议

> 日期：2026-07-22 ｜ Jest 29.7.0 ｜ 数据源：`coverage/coverage-summary.json` + 实跑验证（1972 用例 / 0 失败）
> 关联：`docs/单元测试覆盖率分析-2026-07-22.md`、`docs/driver-sdk-定位与测试策略.md`
> 本文仅给出建议片段，**不改动 jest.config.js**（避免与并行的测试补全工作运行时耦合）。落地时由测试负责人择机合并。

---

## 一、设计原则

1. **只给核心层设防退化门禁**，不给全局设高门槛——driver-sdk 的 templates/examples/testing（1408L 低 ROI 代码，行覆盖 0–12%）会把全局拉红，逼迫团队给代码生成模板刷无意义单测。
2. **阈值 = 实测值 - 2 个百分点**（向下取整），确保当前一定绿，只拦截未来退化。
3. **门禁只盯 lines + functions**，核心层 branches 普遍偏低（models 65.9%、services 64.2%），现阶段设 branches 门禁会误伤。protocols 分支达 83.7%，可额外加 branches。
4. **services 必须排除 self-test 脚本**，否则实测 76% 会让 80% 门禁秒红（详见第三节陷阱）。

---

## 二、实测佐证表（鲜活数据，2026-07-22 19:31 实跑）

验证命令：

```bash
npx jest --testPathPattern=unit --coverage \
  --collectCoverageFrom='src/**/*.ts' \
  --coverageReporters=json-summary
```

结果：77 套件 / 1972 用例 / 0 失败，全局 Lines 59.38%。

| 目标 glob | 实测行覆盖 | 实测分支 | 实测函数 | LOC | **建议阈值(lines)** | 安全余量 |
|-----------|----------:|--------:|--------:|----:|:------------------:|:-------:|
| `src/models/**/*.ts` | **82.97%** | 65.92% | 82.71% | 1415 | **80%** | +2.97 ✓ |
| `src/services/**/*.ts`（含 self-test） | 76.01% | 64.16% | 83.10% | 2997 | — | -3.99 ✗ |
| `src/services/**/*.ts`（**排除 self-test**） | **82.90%** | 69.10% | 88.87% | 2748 | **80%** | +2.90 ✓ |
| `src/decoders/protocols/**/*.ts` | **91.07%** | 83.73% | 93.20% | 1456 | **85%** | +6.07 ✓ |

**核心结论**：models / protocols 直接设阈值安全；**services 的 80% 阈值必须配合"排除 self-test"才成立**——这是本验证步骤抓到的关键陷阱。

## 三、关键陷阱：services 含 self-test 会拉红

`src/services/stage7-self-test.ts` 是一个 249 行的**自检脚本**，行覆盖 0%（它本身就是测试入口，不会被别的测试覆盖）。同类还有 `src/decoders/test-decoder-integration.ts`（72L）、`src/decoders/week3-self-test.ts`（101L）、`src/models/UnifiedDataFormat.ts`（39L，未启用的格式定义）。

| services 口径 | 实测 | 设 80% 阈值 |
|-------------|------|------------|
| 包含 `stage7-self-test.ts` | 76.01% | ❌ 立即拉红 |
| 排除 `stage7-self-test.ts` | 82.90% | ✅ 安全（+2.9） |

`stage7-self-test.ts` 不是产品代码（是 self-test 脚本），把它计入覆盖分母本身就是方法论错误。正确做法是在 `collectCoverageFrom` 源头剔除这类脚本，而非调低阈值。

## 四、推荐的 jest.config.js 配置片段

采用"**collectCoverageFrom 源头净化 + 简单目录前缀阈值**"方案，最干净、最易维护：

```js
// jest.config.js —— 建议追加以下字段
module.exports = {
  // ... 现有配置保留

  // (1) 覆盖率采集范围：从分母里剔除低价值/非产品代码
  collectCoverageFrom: [
    'src/**/*.ts',
    // 排除测试入口与自检脚本（它们本身就是测试，不应被计入覆盖分母）
    '!src/**/*-self-test.ts',
    '!src/**/test-*-integration.ts',
    // 排除 driver-sdk 的代码生成模板 / 示例 / 测试工具自身（低 ROI，见 driver-sdk 定位文档）
    '!src/driver-sdk/templates/**',
    '!src/driver-sdk/examples/**',
    '!src/driver-sdk/testing/**',
    // 排除测试文件本身
    '!src/**/*.test.ts',
    '!src/**/__tests__/**',
    // 排除纯类型声明
    '!src/types/**',
    '!src/**/*.d.ts',
  ],

  // (2) 覆盖率门禁：仅核心层防退化，全局保持低底线
  coverageThreshold: {
    // 全局底线：防止整体崩塌，但不设高门槛（driver-sdk/index 等会拉低）
    global: {
      branches: 50,
      functions: 55,
      lines: 55,
      statements: 55,
    },
    // 核心数据模型层：实测 82.97% → 设 80%
    './src/models/': {
      lines: 80,
      functions: 80,
    },
    // 核心服务层：实测 82.90%（排除 self-test 后）→ 设 80%
    './src/services/': {
      lines: 80,
    },
    // 协议解码器：实测 91.07% / 分支 83.73% → 设 85% / 80%
    './src/decoders/protocols/': {
      lines: 85,
      branches: 80,
    },
  },
};
```

### 为什么这样配

- **collectCoverageFrom 净化分母**：把 self-test 脚本、driver-sdk 模板/示例/测试工具、纯类型声明从统计范围剔除。这些不是产品代码，计入分母会持续误导。净化后 `src/services/` 自然回到 82.9%，`src/models/` 回到 82.8%。
- **coverageThreshold 分桶**：Jest 29 支持 micromatch glob 与目录前缀作为 key（见第五节）。用目录前缀 `'./src/models/'` 比 glob 更可读，且 `collectCoverageFrom` 已做净化，桶内全是产品代码。
- **全局底线 55%**：当前全局 lines 59.38%，55% 给足缓冲，只拦截断崖式退化（例如有人删了大片测试）。
- **不设 driver-sdk 桶**：按 `driver-sdk-定位与测试策略.md` 结论，定位 B 下 driver-sdk 不设门禁。

## 五、Jest coverageThreshold glob paths 正确写法（Jest 29）

Jest 29 的 `coverageThreshold` 支持**目录前缀**与 **micromatch glob** 两种 key 形式。

### 形式 1：目录前缀（推荐，简单可读）

```js
coverageThreshold: {
  './src/models/': { lines: 80 },
  './src/services/': { lines: 80 },
  './src/decoders/protocols/': { lines: 85 },
}
```

- key 必须以 `./` 开头、以 `/` 结尾，匹配该目录下所有被采集文件。
- 依赖 `collectCoverageFrom` 净化桶内文件。

### 形式 2：micromatch glob（精确控制时用）

```js
coverageThreshold: {
  // 匹配 models 下所有 .ts（含子目录）
  './src/models/**/*.ts': { lines: 80 },
  // 排除 self-test：用 extglob 否定
  './src/services/**/!(*-self-test).ts': { lines: 80 },
  // 精确到 protocols
  './src/decoders/protocols/**/*.ts': { lines: 85, branches: 80 },
}
```

- micromatch 默认开启 extglob，`!(*-self-test)` 可排除以 `-self-test` 结尾的文件名。
- **注意**：glob key 的匹配对象是 `collectCoverageFrom` 已采集的文件路径；若 `collectCoverageFrom` 没采集某文件，该文件不会进任何桶。
- 若桶内文件数为 0，Jest 会抛 `ERROR: Coverage for path (...) does not exist`。glob 须确保至少命中一个已采集文件。

### 易错点

| 错误写法 | 问题 |
|---------|------|
| `'src/models'`（无 `./` 前缀） | Jest 不识别，桶不生效 |
| `'./src/models'`（无尾部 `/`） | 被当作文件名匹配，匹配不到目录 |
| `'./src/services/**/*.ts'`（未排除 self-test） | 实测 76%，设 80% 立即拉红 |
| glob 桶为空（collectCoverageFrom 未采集） | Jest 报 coverage path does not exist |
| 用 `branches: 80` 套 models | models 分支仅 65.9%，立即拉红——只对 protocols 设 branches |

### 验证门禁是否生效

落地后跑：

```bash
npx jest --testPathPattern=unit --coverage \
  --coverageReporters=text-summary --json --outputFile=/tmp/jest.json
# 退出码非 0 表示有桶未达阈值；查看 /tmp/jest.json 的 coverageThreshold结果
```

## 六、渐进式引入建议

避免一次性把所有阈值开满导致 CI 抖动，建议分两步：

1. **第一步（低风险）**：先只加 `collectCoverageFrom`（净化分母）和 `coverageThreshold.global`（55% 底线）。这一步不会拉红任何东西，但能让覆盖率报告从此反映真实产品代码基数。跑一次确认报告数值与本文表格一致。
2. **第二步（防退化）**：加入 `./src/models/`、`./src/services/`、`./src/decoders/protocols/` 三个分桶阈值。此时各桶实测均高于阈值 2.9+ 个点，安全。后续若某 PR 使核心层覆盖跌破阈值，CI 失败，强制 review。

两步之间可间隔一个迭代，观察并行测试补全工作（参见 `单元测试覆盖率分析-2026-07-22.md` 的 P0/P1 backlog）对数值的影响——补测只会抬升实测，让阈值余量更大、更安全。

## 七、与 driver-sdk 策略的联动

按 `docs/driver-sdk-定位与测试策略.md` 的推荐（定位 B：内部脚手架）：

- **driver-sdk 不进 coverageThreshold 任何桶**
- driver-sdk 的 templates/examples/testing 通过 `collectCoverageFrom` 的 `!` 排除项从分母剔除（见第四节），它们的不计入会让全局底线更容易满足
- 仅保留 `driver-sdk/tools` 与 `driver-sdk/utils` 在采集范围内（未来定位切 A 时，可对其单独设桶：tools 70%、utils 80%）

若未来定位切换为 A（第三方战略 SDK），追加：

```js
coverageThreshold: {
  // ... 核心层不变
  './src/driver-sdk/tools/': { lines: 70 },   // 当前 6.4%，需先补测
  './src/driver-sdk/utils/': { lines: 80 },   // 当前 55%，需先补测
}
```

注意：这两桶当前实测远低于建议阈值，**切换定位 A 前必须先按 `driver-sdk-定位与测试策略.md` 第五节清单补测**，否则 CI 会立即拉红。
