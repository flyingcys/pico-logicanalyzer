# 覆盖率门禁配置建议

> 日期：2026-07-23 ｜ Jest 29.7.0 ｜ 数据源：`coverage/coverage-summary.json` + 实跑验证  
> 关联：`docs/driver-sdk-定位与测试策略.md`、`jest.config.js`、`src/web-app/vite.config.ts`  
> **本文与当前落地配置对齐**（以 `jest.config.js` 为准）。

---

## 一、设计原则

1. **只给已有稳定测试的关键路径设防退化门禁**，不给全局设高门槛——driver-sdk 工具/模板与零覆盖遗留模块会把全局拉红。
2. **阈值 ≈ 实测值 - 2 个百分点**（向下取整），当前一定绿，只拦截未来退化。
3. **文件级路径阈值优先于目录桶**——遗留模块同一目录内差异大，目录桶易误伤或过松。
4. **web-app 不进 Jest 分母**——由 Vitest 独立门禁（`npm --prefix src/web-app run test:coverage`）。
5. **driver-sdk templates/examples/testing 与 self-test 从分母剔除**。

## 三、关键陷阱

### 1. services / 分母含 self-test

`*-self-test.ts`、`test-*-integration.ts` 是测试入口，行覆盖 0%。必须在 `collectCoverageFrom` 源头剔除，而非调低阈值。

### 2. web-app 进 Jest 分母

`src/web-app/**` 由 Vitest 覆盖；若被 Jest 采集则长期 0%，污染全局与报告。必须 `!src/web-app/**`。

### 3. 过松文件阈值

实测 88% 却设 40% → 删半测试仍绿。新锁路径务必按「实测 - 2」。

### 4. 贴线 100% 入口

`main-html` / `main-vscode` 单测 + 100% 阈值：改入口副作用必须同步补测，否则 CI 立即红（预期行为）。

## 四、落地配置（与 `jest.config.js` 一致）

```js
// jest.config.js 关键字段摘要
module.exports = {
  collectCoverageFrom: [
    'src/**/*.ts',
    '!src/**/*-self-test.ts',
    '!src/**/test-*-integration.ts',
    '!src/driver-sdk/templates/**',
    '!src/driver-sdk/examples/**',
    '!src/driver-sdk/testing/**',
    '!src/web-app/**',
    '!src/**/*.test.ts',
    '!src/**/__tests__/**',
    '!src/types/**',
    '!src/**/*.d.ts',
  ],

  // 无 global：避免低全局值伪装质量
  coverageThreshold: {
    './src/models/LACFileFormat.ts': {
      branches: 78, functions: 85, lines: 85, statements: 85,
    },
    './src/decoders/protocols/I2CDecoder.ts': {
      branches: 55, functions: 80, lines: 80, statements: 80,
    },
    './src/decoders/protocols/SPIDecoder.ts': {
      branches: 70, functions: 80, lines: 80, statements: 80,
    },
    './src/decoders/ChannelMapping.ts': {
      branches: 90, functions: 100, lines: 100, statements: 100,
    },
    './src/decoders/DecoderManager.ts': {
      branches: 39, functions: 56, lines: 69, statements: 68,
    },
    './src/drivers/SigrokAdapter.ts': {
      branches: 65, functions: 63, lines: 68, statements: 69,
    },
    './src/frontend/app/main-html.ts': {
      branches: 100, functions: 100, lines: 100, statements: 100,
    },
    './src/frontend/app/main-vscode.ts': {
      branches: 100, functions: 100, lines: 100, statements: 100,
    },
  },
};
```

### 为什么这样配

- **文件级阈值**：只锁补过行为测试、数值稳定的关键路径。
- **无 global**：全局仍受 driver-sdk tools、零覆盖遗留文件影响；硬 global 只会逼无意义刷测。
- **web-app 排除**：与 Vitest 门禁分工，避免双计/假 0%。
- **配置锁测试**：`tests/unit/quality/QualityGateConfig.test.ts` 断言排除项与关键路径存在。

## 五、Jest coverageThreshold 写法要点

| 写法 | 用途 |
|------|------|
| `'./src/foo/Bar.ts'` | 单文件防退化（当前主用） |
| `'./src/models/'` | 目录前缀桶（目录内覆盖均匀时用） |
| `'./src/models/**/*.ts'` | micromatch glob |

易错：

| 错误 | 问题 |
|------|------|
| key 无 `./` | 桶不生效 |
| 目录 key 无尾部 `/` | 当文件名匹配，常失败 |
| 桶内 0 文件 | `Coverage for path ... does not exist` |
| 阈值远低于实测 | 门禁形同虚设 |

验证：

```bash
npm run test:unit -- --coverage --coverageReporters=text-summary
# 退出码非 0 → 某路径未达阈值
```

## 六、CI 分层

| 步 | 命令 | 作用 |
|----|------|------|
| 单元 + Jest 路径门禁 | `npm run test:unit -- --coverage` | 本文件阈值 |
| 集成 | `npm run test:integration` | 无覆盖率强制 |
| Web | `npm --prefix src/web-app run test:coverage` | Vitest global + `main-web.ts` 100% |
| 压力 | `npm run test:stress` | 不进常规 PR |

常规 PR **不要**裸跑 `npm run test:ci`（会拾 stress/performance/e2e）。

---

## 七、与 driver-sdk 策略

- **driver-sdk 不进 coverageThreshold**
- templates/examples/testing 已从 `collectCoverageFrom` 剔除
- tools/utils 仍采集、不设硬阈值（定位 B：内部脚手架）
- 若切定位 A，先补测再单独加桶（见 `docs/driver-sdk-定位与测试策略.md`）

---

## 八、后续可选项（非当前门禁）

1. 更多 cleanup 关键路径补测后按「实测 - 2」加文件锁。
2. 评估是否排除纯 re-export 的 `**/index.ts`（仅当确认无逻辑）。
3. 修复 unit 套件 open handle 泄漏（不影响阈值数字，影响 worker 退出）。
